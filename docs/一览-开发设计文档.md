# 一览（Overview）开发设计文档

## 1. 文档目标

本文档用于指导 AI 或工程师直接开展“一览（Overview）”首版开发工作。内容覆盖产品范围、技术架构、核心数据模型、页面交互规则、同步方案、AI 集成方案、平台能力、测试与交付要求。

本文档优先级低于 [`“一览”用户要求.md`](./“一览”用户要求.md) 中的明确要求；若两者冲突，以用户原要求为准。

若项目采用无记忆 AI 串行接力开发，本文件不是接手入口。接手顺序应以 [`AI-START-HERE.md`](./AI-START-HERE.md) 为准。

## 2. 首版范围与原则

### 2.1 平台优先级

首版开发与适配顺序：

1. 手机窄屏
2. 平板横竖屏
3. Windows 桌面端
4. Web

本期不纳入开发计划的平台不做实现设计，只保留架构可扩展性。

### 2.2 首版必做能力

- 账号登录
- 事项管理
- 主页表格视图
- 列表页
- AI 页
- 添加/编辑事项页
- 设置页
- 多端实时同步
- 本地提醒通知
- 小组件
- 中英双语

### 2.3 首版不做

- 多人共享/协作
- 附件上传
- 服务端推送消息中心
- 服务端 AI 代理
- 未列入优先级的平台实现

### 2.4 实施原则

- 客户端统一采用 `Uno Platform`
- 服务端统一采用 `ASP.NET Core + PostgreSQL`
- 先完成 MVP 核心链路，再补通知、小组件和多平台适配细节
- 服务端唯一业务职责是同步账号下的事项数据和应用配置
- AI 请求由客户端直接调用用户配置的 OpenAI 兼容接口

## 3. 技术选型

### 3.1 客户端

- 框架：Uno Platform
- 语言：C#
- UI：WinUI 风格 XAML
- 架构：MVVM
- 本地存储：SQLite
- 网络：HttpClient
- 序列化：System.Text.Json
- 本地通知：按平台封装统一通知服务
- 小组件：按平台能力分别实现，客户端统一抽象入口协议
- 国际化：资源文件方案，支持 `zh-CN` 与 `en-US`

### 3.2 服务端

- 框架：ASP.NET Core Web API
- 语言：C#
- 数据库：PostgreSQL
- ORM：Entity Framework Core
- 认证：邮箱验证码/密码
- 同步协议：REST API
- 部署目标：用户自建服务器

### 3.3 AI 接入

- 协议：OpenAI 兼容接口
- 配置项：Base URL、API Key、Model
- 请求链路：客户端直连 AI 接口
- 服务端不参与 AI 转发、不存储对话业务数据之外的推理上下文

## 4. 整体架构

### 4.1 架构分层

- `Presentation`
  - 页面、组件、弹层、适配布局

- `Infrastructure`
  - 本地数据库、远程 API、AI 客户端、通知、小组件、配置存储

- `API`

## 5. 核心业务对象

## 5.1 事项统一模型

所有“日程”“任务”“备忘”统一抽象为 `Item`。

基础字段：

- `Id`
- `UserId`
- `Type`
- `Title`
- `Description`
- `Location`
- `Color`
- `IsImportant`
- `IsCompleted`
- `ReminderConfig`
- `RepeatRule`
- `CreatedAt`
- `UpdatedAt`
- `DeletedAt`
- `LastModifiedAt`
- `SourceDeviceId`

类型字段：

- 日程：
  - `StartAt`
  - `EndAt`
- 任务：
  - `PlannedStartAt`
  - `PlannedEndAt`
  - `DeadlineAt`
- 备忘：
  - `ExpectedDurationMinutes`
  - 可选 `TargetDate`

说明：

- 所有时间统一存储为 UTC，同时保存用户时区标识用于展示与计算。
- `LastModifiedAt` 用于同步冲突覆盖。
- 删除采用软删除，支持同步传播。

## 5.2 配置模型

用户配置 `UserSettings` 首版需要同步全部用户设置。

建议字段：

- `Language`
- `ThemeMode`
- `ThemePreset`
- `WeekStartDay`
- `HomeViewMode`
- `DayPlanStartTime`
- `TimeBlockDurationMinutes`
- `TimeBlockGapMinutes`
- `TimeBlockCount`
- `ListPageDefaultTab`
- `ListPageSortBy`
- `ListPageTheme`
- `AiBaseUrl`
- `AiApiKey`
- `AiModel`
- `SyncServerBaseUrl`
- `NotificationEnabled`
- `WidgetPreferences`

说明：

- 虽然 AI 密钥属于敏感信息，但按当前需求仍纳入同步配置。
- 同步服务器地址与 AI 地址为两套独立配置。

## 5.3 AI 对话记录模型

AI 聊天记录按日存储，按用户选择的日/周/月范围聚合显示。

建议字段：

- `Id`
- `UserId`
- `OccurredOn`
- `Role`
- `Message`
- `CreatedAt`
- `RequestType`
- `LinkedItemIds`

说明：

- 每次对 AI 发起请求时，不附带历史上下文。
- 每次请求只传当前用户消息和系统提示词。
- 对话记录用于展示与追踪，不用于构造下次模型上下文。

- 列表页
- AI 页
- 添加页
- 设置页

手机端采用底部导航栏；平板与桌面端可根据宽度切换为更适合的布局，但页面逻辑保持一致。

## 6.2 时间选择组件

### 输入参数

- `SelectionMode`：日 / 周 / 月
- `InitialDate`
- `WeekStartDay`

### 结构

- 顶栏：跳转箭头、年月展示、确认按钮
- 主体：月份表格

### 行为规则

- 可点击日期格、周格、月格
- 选中后显示小圆圈
- 当传回目标为周或月时，点击日期格需要自动映射到所属周格或月格
- 点击确认返回目标时间段
- 左右滑动切换前后月份
- 周起始日基于用户设置动态计算

## 6.3 主页（表格页）

### 顶栏

- 居中显示当前时间段名称
- 点击标题展开时间选择组件
- 左右箭头切换上一/下一时间段

### 主体表格

列规则：

- 每列代表一天
- 窄屏默认显示一周
- 宽屏可切换显示一个月

行规则：

- 行表示用户设置的时间块
- 缺省值：
  - 起始时间 `08:00`
  - 时长 `60` 分钟
  - 间隔 `0` 分钟
  - 数量 `12`

单元格规则：

- 单元格固定为圆角正方形
- 表格高度超出可滚动
- 表格宽度切换与滑动手势联动

事项渲染规则：

- 仅“日程”和“任务”渲染在主页表格上方
- 按真实时间比例跨越多个时间块显示
- 需要支持重叠显示

布局算法要求：

- 先根据用户设置生成当天时间块列表
- 将每个事项时间区间映射到当日纵向分钟轴
- 事项顶部偏移 = `(事项开始时间 - 当日规划开始时间) / 时间块总分钟数 * 表格总高度`
- 事项高度 = `事项实际持续分钟数 / 时间块总分钟数 * 表格总高度`
- 跨天事项在主页只渲染当前列可见部分
- 当事项早于规划起始时间或晚于规划结束时间时，只渲染可见区间
- 任务与日程视觉层级一致，不做固定优先级覆盖

事项重叠透明度规则：

- 遍历单个事项所覆盖的时间范围
- 找出该事项所经历的最大重叠数 `n`
- 当前事项透明度设为 `100 / n` 百分比

点击命中规则：

1. 若涉及事项都存在某段独占区间，则命中起始时间最早的事项
2. 若仅一个事项被其他事项完全包裹，则命中被包裹事项
3. 若多个事项都被完全包裹，则命中这些事项中起始时间最早者

命中实现要求：

- 命中计算基于点击点对应时间坐标和当天列坐标
- 不允许仅按绘制顺序命中
- 算法必须可单元测试，不能写成纯 UI 事件分支

手势规则：

- 长按空白单元格：打开添加页，并自动填充对应起始日期时间
- 长按事项：打开该事项编辑页
- 点击事项：打开事项详情组件
- 左右滑动表格：等同于切换上一/下一时间段

### 事项详情组件

### AI 能力边界

首版仅支持：

- 自然语言新建事项

首版不支持：

- 自动全局规划

### AI 请求规则

- 客户端直接请求用户配置的 AI 接口

说明：

- “根据软件里的事项信息回答问题” 需要客户端先从本地数据库读取相关事项，再组织为结构化上下文提供给模型。

## 7. 数据同步设计

## 7.1 同步目标

同步范围：

- 当前账号下全部事项
- 当前账号下全部应用配置

不同步范围：

- 附件
- 多人共享数据
- 服务端 AI 状态

## 7.2 同步模式

采用“本地优先 + 最后修改时间覆盖”。

规则：

- 本地编辑先写入本地数据库
- 本地生成变更记录并标记待同步
- 后台同步服务定期推送本地变更并拉取远端变更
- 冲突时以 `LastModifiedAt` 较新的记录覆盖较旧记录

## 7.3 删除策略

- 客户端删除采用软删除
- 删除记录同步到服务端
- 服务端向其他设备传播删除状态

## 7.4 首次登录与增量同步

首次登录：

- 拉取用户全部事项与设置
- 建立本地数据库初始快照

后续同步：

- 基于最后同步时间拉取变更
- 上传本地未同步变更

## 7.5 同步异常处理

- 网络失败：保留本地待同步队列并重试
- 鉴权失败：标记账号失效并要求重新登录
- 数据反序列化失败：记录错误日志并跳过异常项
- 局部写入失败：回滚当前事务，不影响已完成同步批次

## 8. 服务端设计

## 8.1 服务端职责边界

服务端仅处理：

- 用户注册与登录
- 事项 CRUD
- 配置 CRUD
- 事项与配置同步

服务端不处理：

- AI 推理
- AI 转发
- 推送消息中心
- 多人协作

## 8.2 服务端模块

- `Auth`
- `Items`
- `Settings`
- `Sync`

## 8.3 API 设计建议

认证接口：

- `POST /api/auth/register`
- `POST /api/auth/login`
- `POST /api/auth/send-verification-code`
- `POST /api/auth/refresh`

事项接口：

- `GET /api/items`
- `POST /api/items`
- `PUT /api/items/{id}`
- `DELETE /api/items/{id}`

配置接口：

- `GET /api/settings`
- `PUT /api/settings`

同步接口：

- `GET /api/sync/pull?since=...`
- `POST /api/sync/push`

说明：

- `pull` 返回事项变更与配置变更
- `push` 接收客户端待同步变更集
- 所有接口基于当前登录用户进行数据隔离

### 8.3.1 认证契约建议

`POST /api/auth/register`

请求：

```json
{
  "email": "user@example.com",
  "password": "plain-or-prehashed-by-policy",
  "verificationCode": "123456"
}
```

响应：

```json
{
  "userId": "uuid",
  "accessToken": "jwt",
  "refreshToken": "token",
  "expiresAt": "2026-03-13T12:00:00Z"
}
```

`POST /api/auth/login`

请求：

```json
{
  "email": "user@example.com",
  "password": "plain"
}
```

响应与注册成功一致。

### 8.3.2 事项契约建议

`GET /api/items`

查询参数：

- `updatedAfter`
- `includeDeleted`
- `type`

`POST /api/items`

请求：

```json
{
  "id": "uuid",
  "type": "task",
  "title": "Prepare slides",
  "description": "for Monday review",
  "location": "Room A",
  "color": "#4A90E2",
  "isImportant": true,
  "isCompleted": false,
  "plannedStartAt": "2026-03-14T01:00:00Z",
  "plannedEndAt": "2026-03-14T03:00:00Z",
  "deadlineAt": "2026-03-15T10:00:00Z",
  "reminderConfig": {},
  "repeatRule": {},
  "lastModifiedAt": "2026-03-13T08:00:00Z",
  "sourceDeviceId": "device-1"
}
```

### 8.3.3 同步契约建议

`GET /api/sync/pull?since=...`

响应：

```json
{
  "serverTime": "2026-03-13T08:00:00Z",
  "items": [
    {
      "id": "uuid",
      "type": "schedule",
      "title": "Demo",
      "deletedAt": null,
      "lastModifiedAt": "2026-03-13T07:59:00Z"
    }
  ],
  "settings": {
    "weekStartDay": "Monday",
    "language": "zh-CN"
  }
}
```

`POST /api/sync/push`

请求：

```json
{
  "deviceId": "device-1",
  "lastKnownServerTime": "2026-03-13T07:50:00Z",
  "itemChanges": [
    {
      "operation": "upsert",
      "item": {
        "id": "uuid",
        "title": "Demo",
        "lastModifiedAt": "2026-03-13T08:01:00Z"
      }
    }
  ],
  "settingsChange": {
    "lastModifiedAt": "2026-03-13T08:01:10Z",
    "value": {
      "weekStartDay": "Sunday"
    }
  }
}
```

响应：

```json
{
  "accepted": true,
  "serverTime": "2026-03-13T08:01:20Z",
  "conflicts": []
}
```

实现要求：

- `push` 必须幂等
- 服务端必须拒绝当前用户无权修改的记录
- 若服务端记录更新更晚，则返回冲突信息供客户端覆盖本地

## 8.4 数据表建议

最少包含：

- `users`
- `verification_codes`
- `items`
- `user_settings`
- `sync_change_logs`

## 9. AI 设计

## 9.1 AI 系统提示词职责

系统提示词需要约束模型：

- 识别用户要新增、删除、查询哪类事项
- 以结构化格式返回解析结果
- 不虚构不存在的数据
- 当信息不足时要求用户补充必要字段

## 9.2 AI 调用流程

1. 用户输入自然语言
2. 客户端识别请求类型
3. 若是问答，先检索本地事项数据并生成摘要
4. 组装系统提示词、当前用户消息、必要上下文
5. 调用 OpenAI 兼容接口
6. 解析模型返回结果
7. 如涉及事项写操作，则先更新本地，再进入同步队列
8. 保存聊天记录

## 9.3 AI 结构化返回建议

建议要求模型输出 JSON，至少覆盖：

- `intent`
- `itemType`
- `title`
- `description`
- `startAt`
- `endAt`
- `deadlineAt`
- `location`
- `importance`
- `reminder`
- `repeatRule`
- `confidence`
- `answer`

推荐响应示例：

```json
{
  "intent": "create_item",
  "itemType": "task",
  "title": "Buy groceries",
  "description": "milk and bread",
  "startAt": "2026-03-13T10:00:00+08:00",
  "endAt": "2026-03-13T11:00:00+08:00",
  "deadlineAt": null,
  "location": "Supermarket",
  "importance": false,
  "reminder": {
    "enabled": true,
    "minutesBefore": 30
  },
  "repeatRule": null,
  "confidence": 0.93,
  "answer": "已为你创建今天上午 10 点到 11 点的购物任务。"
}
```

解析要求：

- 客户端先校验 JSON 结构，再决定是否执行写操作
- 当 `confidence` 低于阈值时，不直接写入，转为向用户追问
- 当关键字段缺失时，不允许默认猜测时间或类型
- 所有 AI 写操作都要通过与手工创建相同的领域校验

### 9.4 AI 意图枚举

首版固定以下意图：

- `create_item`
- `delete_item`
- `query_items`
- `answer_question`
- `clarify`

不允许 AI 输出首版之外的新动作类型。

说明：

- 当 `intent` 为问答时，主要使用 `answer`
- 当 `intent` 为新建/删除/查询时，客户端按结构化字段执行

## 10. 通知与小组件

## 10.1 通知

首版只做本地提醒通知。

要求：

- 根据事项提醒配置在本地调度通知
- 支持事项变更后重新安排通知
- 支持删除事项后取消通知
- 支持跨平台统一通知抽象接口

## 10.2 小组件

首版纳入开发计划的小组件：

- 大组件：主页表格
- 大组件：列表页
- 小组件：AI 交互入口
- 小组件：新建事项入口

设计原则：

- 统一由客户端输出小组件所需轻量数据快照
- 平台实现层负责读取快照并渲染
- 点击小组件后跳转到应用对应页面或动作

## 11. 国际化设计

首版支持：

- 简体中文
- 英文

要求：

- 所有界面文案进入资源文件
- 日期、时间、星期、月份名称按当前语言本地化
- 顶栏标题和时间范围名称支持本地化格式化

## 12. 安全与配置

## 12.1 认证安全

- 密码服务端加盐哈希存储
- 登录令牌采用短期访问令牌 + 刷新令牌
- 客户端敏感配置本地加密存储

## 12.2 配置安全

- 虽然 AI API Key 要同步，但客户端与服务端仍需尽量加密存储
- 所有服务端接口必须鉴权
- 同步接口必须校验当前用户的数据归属

## 15. AI 开发执行要求

为避免 AI 在实施时偏离需求，执行时必须遵守以下规则：

- 不能擅自扩展多人协作、附件、服务端 AI 等超范围能力
- 页面和交互必须优先满足需求文档中的明确规则
- 时间选择、主页事项重叠、命中逻辑必须严格实现，不可简化为普通日历
- 服务端职责不能越界到 AI 转发和推送中心
- 所有新增代码必须围绕跨平台一致性与后续适配性设计
- 所有用户可见文案必须支持中英双语
- 所有本地状态都要考虑同步与离线回放

## 16. 非功能要求

### 16.1 性能

- 手机端主页首屏进入目标时间不超过 2 秒
- 主页在周视图下渲染 200 条事项时仍应保持可滚动
- 列表页在 1000 条事项规模下滚动不得明显卡顿

### 16.2 可维护性

- 领域规则必须可单元测试
- 页面层业务分支不得直接依赖平台实现
- 所有跨平台能力必须经由接口抽象注入

### 16.3 观测性

- 同步失败、AI 解析失败、通知调度失败必须有日志
- 首版至少保留本地日志与服务端结构化日志
