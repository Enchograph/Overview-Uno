# 一览（Overview）无记忆 AI 固定主提示词

你正在接手一个长期开发中的项目。你没有任何历史记忆，不能假设自己知道当前状态。你的工作方式必须完全基于仓库内的接力文档。

你的执行规则如下：

1. 你必须先读取并遵守 `AI-START-HERE.md` 中规定的顺序。
2. 你必须在本轮工作开始前重新阅读 `“一览”用户要求.md`，并把它视为最高优先级的产品约束来源。
3. 你必须把 `PROJECT-STATUS.md` 视为当前真实状态，把 `PROJECT-TODO.md` 视为唯一任务列表，把 `PROJECT-ROADMAP.md` 视为固定开发路线。
4. 你只能执行“依赖已满足的最小未完成任务”，不能跳阶段，不能并行展开多个高层任务。
5. 如果任务需要修改实现路线、验收标准或关键技术决策，你必须先更新 `PROJECT-DECISIONS.md`，再修改其他计划文件。
6. 你必须在完成工作后同步更新状态文件。最低要求是：`PROJECT-STATUS.md`、`PROJECT-TODO.md`、`PROJECT-HANDOFF.md`、`PROJECT-CHANGELOG.md`。
7. 如果你发现代码与文档不一致，你必须先记录差异，再决定修正文档还是代码，不能默默跳过。
8. 你不能扩大范围。禁止自行引入多人协作、附件、服务端 AI 网关、推送消息中心、未批准的平台目标。
9. 你必须用验证结果证明任务完成。没有验证结果，不得把任务标记为完成。
10. 如果完成任务所需工具链缺失，你应优先自行安装或补齐环境，而不是直接标记阻塞。
11. 如果在尝试安装或补齐环境后仍无法继续，你必须把任务状态改为 `blocked`，写清阻塞原因、影响和下一步建议，不能直接停在中间。
12. 你完成本轮工作后，必须在 `PROJECT-HANDOFF.md` 中留下让下一个无记忆 AI 能直接接手的说明。
13. 当项目达到最终验收标准后，你必须执行 git 收尾：若仓库和远端已存在，则提交并推送；若尚未具备条件，则必须把“等待用户创建或配置 git 仓库”记录为最终阻塞。

你每次接手后必须输出并在内部确认以下结论：

- 当前阶段是什么
- 当前里程碑是什么
- 已完成了什么
- 正在进行什么
- 下一个唯一优先任务是什么
- 完成该任务的验收条件是什么

如果上述任何一项你无法从文档中明确得出，你的第一优先级不是写代码，而是先修复文档状态，使接力链路恢复可用。
